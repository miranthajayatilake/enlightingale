## 1. The One-Sentence Pitch

> The user should only ever describe what they're curious about — the system figures out the name, finds the sources, and hands them a fully-populated Canvas with the Mentor already open and ready to walk them through it.

---

## 2. Why This, Why Now

v0.3 made the creation wizard richer (description, level, starting points) and the Research Engine accessible from the Canvas. But three friction points remain:

1. **The user still has to name their Muse.** Names are boilerplate — nobody knows the perfect title for an idea they haven't explored yet. The system has the description; it should derive the name.

2. **The first Canvas is thin.** It's built from the description alone (and any user-provided starting points), but the Research Agent hasn't run yet. The Canvas is essentially a two-paragraph overview. The user has to manually trigger a research pass and then rebuild the Canvas to get anything substantive. The "land on a beautiful Canvas" promise isn't met on day one.

3. **The Mentor doesn't greet you.** The Canvas loads and the Mentor pane is collapsed. The single most powerful feature — a narrated walkthrough — requires the user to notice and click a strip on the right edge.

v0.3.1 fixes all three and rewires the creation pipeline so that by the time the user sees their Canvas for the first time, it has already been enriched by a Research Agent pass.

---

## 3. Goals & Non-Goals

### 3.1 Goals
- **G1.** Remove the name field from Muse creation. The name is generated by the backend from the description.
- **G2.** Interpret the user's description to identify the *learning focus* — even when the description is a problem statement rather than a topic name.
- **G3.** Research Agent runs automatically as the first step of the creation pipeline. The Canvas is built after the Agent pass completes (so it reflects real research, not just the description).
- **G4.** Note titles in the "starting points" step are auto-generated from note content — no title field required.
- **G5.** The Mentor pane opens automatically the first time a Muse's Canvas is presented.

### 3.2 Non-Goals
- No change to the Canvas structure or section types.
- No change to the Research Agent's search/evaluate/curate logic.
- No multi-user or auth work.
- No change to the KL auto-rebuild behaviour (that stays as-is from v0.3).

---

## 4. User Stories

**US1 — Frictionless creation**
> As a user, I describe what I want to explore in plain language. I don't have to think of a title — a good name appears when my Muse is ready.

**US2 — Problem-first descriptions**
> As a user, I say "I've been struggling to understand why my startup's retention is poor." The system infers I want to learn about *user retention strategy* and builds a focused Muse on that — not on "startup retention problems."

**US3 — Rich first Canvas**
> I finish the creation wizard and see a build progress screen. When the Canvas finally appears, it already has sources from the web in it — it's substantive, not a stub.

**US4 — Mentor-first landing**
> The first time my Canvas loads, the Mentor pane is open and ready. I can immediately start the Guided Tour or just chat.

**US5 — Effortless notes**
> I paste a chunk of text into the Note adder during creation. I don't have to think of a title — it's generated for me.

## 5. Feature Specifications

### 5.1 Description-First Muse Creation (no name field)

**Current flow:** Step 1 = Name, Step 2 = Description + Level, Step 3 = Knowledge Level, Step 4 = Starting Points.

**New flow:** Step 1 = Description + Level (merged), Step 2 = Starting Points (optional). The name is never asked for.

#### Backend: Name + Focus generation

A new helper `services/muse/interpreter.py` with a single async function:

```python
async def interpret_description(description: str) -> InterpretedMuse:
    """
    Given a free-text description, return:
    - name: a short (2–5 word) Muse name, title-cased, no punctuation
    - focus: a one-sentence learning focus statement for the Research Agent
    """
```

Called via a single Claude prompt (cheap, fast — `claude-haiku-4-5-20251001` is fine here):

```
The user wants to create a personal learning space. They described it as:
"{{description}}"

Your job:
1. Identify what they actually want to *learn about* — even if they described a problem, a goal,
   or a situation rather than a topic. Think: "someone studying this would be studying ___."
2. Return a JSON object:
   {
     "name": "2–5 word title, title-cased, no trailing punctuation",
     "focus": "One sentence: what this Muse should help the user understand, framed as a learning goal."
   }

Rules:
- name must be a noun phrase (not a question, not a sentence)
- focus should be specific enough to guide a researcher — not "learn about X" but "understand the
  mechanisms behind X and how practitioners approach Y"
- If the description is already a clear topic, reflect it directly
- If it's a problem or situation, abstract to the underlying domain of knowledge
```

`InterpretedMuse` is a Pydantic model: `{ name: str, focus: str }`.

The `focus` is stored on the `Muse` model as a new column: `muse.research_focus: str | None`. It is passed to the Research Agent planner instead of (or in addition to) the Muse description, giving the agent a sharper brief.

#### Frontend: Step 1 redesign

Remove the name `<Input>`. The single textarea is the full description (placeholder: *"What do you want to explore? Describe it in your own words — a topic, a problem, a question, anything."*). Below it, the knowledge level selector stays.

The Continue button label on Step 1 changes to **"Let's go →"**. No name preview is shown during creation — the user will see the name when the Canvas loads.

The Muse title in `Sidebar.tsx` and page headings is populated from the server response. Since creation is async (name is generated server-side), the `POST /api/muses` response already includes the generated name.

#### API change

`POST /api/muses` body changes:

```python
class MuseCreate(BaseModel):
    description: str
    knowledge_level: str = "beginner"
    # `name` removed — generated server-side
```

`create_muse` in `api/muses.py`:
1. Calls `await interpret_description(body.description)` → `interpreted`
2. Sets `muse.name = interpreted.name`, `muse.research_focus = interpreted.focus`
3. Saves and returns the Muse as before

`NewMuse.tsx` stops sending `name` and reads `muse.name` from the response for display.

### 5.2 Research-First Creation Pipeline

**Current pipeline on Muse creation:**
```
POST /api/muses → KL build (from description) → Canvas build
```

**New pipeline:**
```
POST /api/muses → Research Agent (focused on research_focus) → KL build → Canvas build
```

The KL build is no longer enqueued directly on creation. Instead, the Research Agent runs first, and on its completion it triggers `maybe_enqueue_kl_build` (the same mechanism already used when resources are approved).

#### Backend changes

`api/muses.py — create_muse`:
- Remove the `maybe_enqueue_kl_build` call.
- Instead, create a `BackgroundJob(job_type="research_agent")`, set `muse.agent_status = "running"`, and enqueue `run_research_agent(muse_id, job_id, focus=muse.research_focus)`.

`services/research_agent/agent.py — run()`:
- On completion, after setting `muse.agent_status = "complete"` and approving all gathered resources, call `maybe_enqueue_kl_build(muse_id, redis_conn, session)`.
- This is the trigger that starts the KL → Canvas chain.

The existing `maybe_enqueue_kl_build` already handles the KL → Canvas chain (KL completes → Canvas builds). No changes needed there.

#### CanvasBuildStages: 4-stage pipeline with real progress feedback

The build progress UI gains a new first stage, and every stage shows real progress — not just a sweep animation. The user should always know exactly what the system is doing.

| Stage id | Label | Active when | Progress source |
|---|---|---|---|
| `research` | Searching the web | `muse.agent_status === 'running'` | WebSocket job events (0–100 + subtopic names) |
| `sources` | Reading your sources | agent done, resources processing | resource status polling |
| `knowledge` | Building knowledge layer | kl.status === 'building' | BackgroundJob.progress via polling |
| `canvas` | Composing your Canvas | canvas.status === 'building' | BackgroundJob.progress via polling |

`sources` stage only shown if `muse.resource_count > 0` after agent completes (same as current logic).

##### Research stage — rich live feedback

The Research Agent already broadcasts granular events via WebSocket (`/ws/jobs/{job_id}`): numeric progress (0–100), a `step` label, and per-subtopic `searching`/`found` events. The existing `useAgentStatus` hook in `Research.tsx` consumes all of this. `CanvasBuildStages` should use the same data.

`CanvasBuildStages` needs access to the live agent job. Pass the agent `jobId` as a prop from `Overview.tsx` (available from the Muse's latest agent job query, or from the job returned when the Muse was created). Internally, `CanvasBuildStages` calls `useAgentStatus(museId, agentStatus)` during the `research` stage to get `latestEvent`, `subtopics`, and `searching`.

What the user sees during the research stage:
- A real percentage fill bar (0 → 100%) driven by `latestEvent.progress`
- The current step label beneath the bar: e.g. *"Evaluating sources…"*, *"Curating results…"*
- A live list of subtopics being searched, appearing one by one as the agent opens each:
  ```
  ✓  User retention frameworks        (done)
  ⟳  Churn analysis techniques        (searching…)
  ○  Product-led growth strategies    (queued)
  ```
  Max 5 rows, overflow hidden. Subtopics come from `subtopics` + `searching` state in `useAgentStatus`.

##### Knowledge + Canvas stages — numeric progress bar

For `knowledge` and `canvas` stages, `BackgroundJob.progress` (0–100) is available via the existing polling on the job query. Show a real percentage fill bar. The step label comes from `BackgroundJob.status_message` (already updated at each checkpoint in `builder.py` and `canvas/builder.py`).

What the user sees during knowledge/canvas stages:
- Real fill bar (e.g. 45% → "Extracting concepts…", 80% → "Building glossary…")

##### Sources stage — resource count feedback

No job progress available here (multiple parallel resource jobs). Show:
- *"Processing N sources…"* where N counts from polling `resources` query
- The existing sweep animation bar (no numeric progress available)

##### Overall warm messages (bottom of card)

| Active stage | Message |
|---|---|
| `research` | *"Scouting the web for the best sources on **{muse.name}**…"* |
| `sources` | *"Making sense of everything collected…"* |
| `knowledge` | *"Your Mentor is learning about **{muse.name}**…"* |
| `canvas` | *"Almost there. Writing your visual overview…"* |

#### Resource approval in the agent pipeline

Currently agent-gathered resources have `approved=False` and appear in `ResourceReviewList` for the user to approve manually. In the creation pipeline this is awkward — the user would land on a "Building" screen and never see the review list.

For the **creation-time first pass only**, all resources gathered by the agent are auto-approved: `approved=True`. This matches the intent — the user said "go research this for me," and blocking on review defeats the purpose.

For subsequent agent passes from the Research tab, the existing approve/review flow stays as-is. The distinction is: `is_first_pass = (muse.agent_status_was_never_complete)` — i.e., the very first time the agent runs for this Muse (tracked by checking `muse.agent_status` before the run: if it has never been `complete` before, auto-approve).

Implementation: pass `auto_approve: bool` to `run_research_agent`. Set `auto_approve=True` from `create_muse`, `False` from `api/research_agent.py — run_agent`.

### 5.3 Auto-Generated Note Titles

**Current:** `NoteAdder` has two fields: Title and Content.

**New:** `NoteAdder` has one field: Content only. Title field is removed.

The title is generated server-side when the resource is created. In `api/resources.py — create_resource` (for `source_type="text"`):
- If no `title` is provided, derive a title from content:
  - Take the first non-empty line, strip markdown, truncate to 80 chars.
  - This is a deterministic rule (no LLM call needed — good enough for a note title).
- `title` becomes optional in `ResourceCreate`.

In `NewMuse.tsx — NoteAdder`:
- Remove the title `<Input>`.
- The `StartingPoint` type: `noteTitle` becomes optional (or removed — the label shown in the "Added" list can be derived the same way client-side for display).

### 5.4 Mentor Pane Opens on First Canvas

When the Canvas is rendered for the first time for a Muse, the Mentor pane should be open (expanded) automatically.

**"First time" definition:** The Canvas has just transitioned from `building → ready` in the current session (not on every page load/refresh). This avoids the Mentor forcing itself open every time the user navigates back to the Canvas tab.

**Implementation:**
- `Overview.tsx` already tracks `canvasStatus`. When it transitions to `'ready'` and `canvas` goes from null/building to a ready `MuseCanvas` object, call `requestMentorOpen()` from `useMentorPaneStore` — the same mechanism the Canvas end-CTA already uses.
- Gate this on a ref (`mentorOpenedForCanvas`) so it only fires once per session, not on every refetch.

## 12. Key Design Decisions

**KD1 — No name preview step.** Showing the AI-generated name mid-creation and asking the user to confirm adds a round-trip and friction. The name is good enough to start. Users can rename later (Phase 2 settings). Trust the system.

**KD2 — Research Agent auto-approves on first pass only.** The review flow exists because the agent can pull tangentially-related sources. But on creation the user explicitly asked the system to "go research this" — blocking on review defeats the point. For subsequent passes the user may want to curate, so the review flow stays.

**KD3 — Canvas is built after Research Agent, not before.** A Canvas built from a description alone is thin and generic. Waiting 1–2 extra minutes for a Canvas that reflects real research is worth it. The 4-stage build animation makes the wait legible.

**KD4 — Note title derived from first line, not LLM.** A deterministic text rule (first non-empty line, 80 chars) is fast, free, and good enough for a note. LLM title generation for every note create would add latency and cost.

**KD5 — `research_focus` stored on the Muse.** It's the system's interpretation of what the user wants to learn. It's reusable — subsequent focused Research Agent passes from the Canvas gaps use it as the base context. It's also useful for the Mentor's system prompt (future).
