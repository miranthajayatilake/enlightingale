## Backend Findings

### High

**B-H1** `backend/services/knowledge/builder.py:228`
Bare `except Exception: pass` in the post-build unembedded-resource re-trigger block.
If `maybe_enqueue_kl_build` throws (Redis down, DB error), the failure is silently swallowed with no log. Resources that finished processing while the KL was running are permanently abandoned.
→ Replace with `logger.error(...)` at minimum.

**B-H2** `backend/api/research_agent.py:68–82`
`GET /agent/results` has no `response_model=` and returns a raw dict including a bare ORM object for `job`.
→ Add a typed response model and serialize `job` through `JobRead`.

**B-H3** `backend/api/research_agent.py:97–99`
`approve_all` sets `muse.resource_count` to a full table count of ALL resources, including unapproved agent resources.
→ Count only approved resources, or skip updating resource_count here (it's updated correctly by individual approve/add calls).

### Medium

**B-M1** `backend/services/knowledge/autorebuild.py:19–32`
`KnowledgeLayer.status` is set to `"building"` and committed before `enqueue_job` is called. If Redis is unavailable after the DB commit, the KL is stuck at `"building"` forever with no running job.
→ Wrap `enqueue_job` in try/except; set job+KL to `"failed"` if enqueue fails.

**B-M2** `backend/services/knowledge/builder.py:59–61`
If the Muse no longer exists, the function returns early without setting `BackgroundJob.status = "failed"`. The job record is stuck in `"queued"` state.
→ Set job and KL to `"failed"` on the early return.

**B-M3** `backend/api/resources.py:183–184`
`delete_resource` triggers a KL rebuild even when the deleted resource was never embedded (e.g., still `status="pending"`). Deleting a pending resource does not change the knowledge layer state.
→ Gate on `resource.embedded` (or `resource.status == "ready"`).

**B-M4** `backend/services/knowledge/builder.py:185`
Canvas first-build guard comment says "no Canvas exists yet" but the condition `canvas.status not in ("ready", "building")` also fires for `"failed"` and `"idle"` states, which is actually correct behaviour. Comment is misleading.
→ Update the comment to accurately describe the condition.

### Low

**B-L1** `backend/services/knowledge/autorebuild.py`
No logging anywhere — no trace of debounce-skips or successful enqueues. Makes debugging KL trigger issues impossible.
→ Add `logger` and log both paths.

**B-L2** `backend/api/research_agent.py:13–16`
No blank line between `AgentRunBody` class definition and `router` assignment (PEP 8).

**B-L3** `backend/api/resources.py:153–161`
`update_resource` triggers KL rebuild on approval even if resource is `status="pending"` (not yet processed/embedded). Should check `resource.status == "ready"`.
→ Gate KL rebuild in `update_resource` on `resource.status == "ready"`.

**B-L4** `backend/services/research_agent/planner.py:19–22`
`focus` string is inserted directly into the LLM prompt with no sanitization. Fine for Phase 1 (single user), but note for Phase 2 multi-user auth landing.

## Frontend Findings

### High

**F-H1** `frontend/src/pages/muse/Research.tsx:54–59`
`runAgent.onSuccess` only invalidates `['muse', id]` and `['muses']`. Does not invalidate `['canvas', id]`, `['knowledge', id]`, or `['agent-results', id]`. After a new agent pass, the canvas stale banner won't appear and the agent results panel will show stale coverage/gaps from the previous run.
→ Invalidate all four query keys in `onSuccess`.

**F-H2** `frontend/src/features/canvas/Canvas.tsx:56–62` / `frontend/src/features/canvas/sections/GapsSection.tsx:21–26`
`runFocusedAgent.isPending` is never passed to `GapsSection`. All "Research this →" buttons are always enabled. A user can click multiple gaps rapidly, launching concurrent agent jobs.
→ Add `focusedResearchPending?: boolean` to `SectionProps`, pass `runFocusedAgent.isPending` from `Canvas.tsx`, disable buttons in `GapsSection`.

**F-H3** `frontend/src/features/chat/ChatPanel.tsx:191`
`handleSave` has an empty `catch` block — save failures are silently swallowed. The button returns to "Save to resources" with no feedback, and the user may keep retrying.
→ Set a transient error state and show a brief failure message.

### Medium

**F-M1** `frontend/src/pages/muse/Research.tsx:54–59`
`runAgent.isError` is never rendered. If `POST /agent/run` fails (Redis down, 500), no feedback is shown.
→ Show an inline error message when `runAgent.isError`.

**F-M2** `frontend/src/features/canvas/Canvas.tsx:150–157`
The `⋯` options menu closes but shows no loading feedback when "Rebuild Knowledge Layer" or "Run Research Agent again" are clicked.
→ Show pending state for these mutations in the header area.

**F-M3** `frontend/src/features/canvas/Canvas.tsx:150–157`
Options dropdown `useEffect` only listens for `mousedown` outside the menu; does not close on Escape key.
→ Add `keydown` listener for `Escape`.

**F-M4** `frontend/src/pages/muse/Research.tsx:152–159`
`deleteResource.isPending` is a single shared state — deleting any resource disables ALL Remove buttons simultaneously.
→ Track the specific resource ID being deleted.

**F-M5** `frontend/src/pages/muse/Overview.tsx:64–68` and `76–80`
`buildCanvas` and `buildKl` (full mutation result objects) are in `useEffect` dependency arrays. The mutation object is recreated on every state change, causing the effects to re-evaluate on every mutation state transition. The `ref` guard prevents double-firing, but it's unnecessarily noisy.
→ Use `buildCanvas.mutate` and `buildCanvas.isPending` as deps instead of the whole object.

**F-M6** `frontend/src/pages/NewMuse.tsx:PdfAdder`
`PdfAdder` displays file size but does not validate against the backend's 50 MB upload limit. A user can add a 200 MB PDF; the upload will fail with an opaque generic error.
→ Add size check in `handleFile` and show an inline error.

**F-M7** `frontend/src/pages/muse/Research.tsx:288`
Hardcoded `style={{ height: 520 }}` on the chat container. Will overflow or feel cramped on short viewports.
→ Replace with Tailwind `min-h-[400px] max-h-[600px]`.

**F-M8** `frontend/src/features/chat/ChatPanel.tsx`
When switching between chat sessions, `displayMessages` is immediately cleared and the component shows the empty "Ask anything" state while `sessionData` loads. No loading indicator during the transition.
→ Show a `Spinner` while `sessionData.isFetching` and `displayMessages` is empty.

**F-M9** `frontend/src/pages/muse/Research.tsx:47–51`
`agent-results` query is not invalidated when a new agent run starts. When the run completes and the query re-enables, it may serve the old run's coverage summary and gaps (60 s stale time).
→ Add `queryClient.invalidateQueries({ queryKey: ['agent-results', muse.id] })` to `runAgent.onSuccess`.

### Low

**F-L1** `frontend/src/features/canvas/Canvas.tsx:CanvasEndCTA`
"Send the Research Agent" button only navigates to the Research tab. Label implies an action ("Send") but is actually navigation. Consider renaming or wiring it to trigger a broad pass directly.

**F-L2** `frontend/src/features/voice/MentorPane.tsx` (pre-existing)
Transcript entries use array index as React key. Stable IDs would be safer.

**F-L3** `frontend/src/features/chat/ChatPanel.tsx:236`
Icon-only delete session button uses `title=` but not `aria-label=`. Screen readers may not announce `title`.
→ Add `aria-label="Delete this conversation"`.

**F-L4** `frontend/src/pages/NewMuse.tsx:Step4 tab buttons`
Tab buttons have no `focus-visible:ring` class. Not keyboard-navigable with visible focus.
→ Add `focus-visible:ring-2 focus-visible:ring-accent/50 outline-none`.

**F-L5** `frontend/src/pages/muse/Research.tsx:SourcesSection empty state`
"No sources yet" empty state does not mention agent-gathered resources awaiting review. User who just ran the Research Agent will see "No sources yet" and be confused.
→ Add a hint about approving agent resources.

**F-L6** `frontend/src/features/canvas/sections/GapsSection.tsx`
Empty `items` array renders the section title + subtitle with an empty list, which looks like an error.
→ Return `null` when `items.length === 0`.

**F-L7** `frontend/src/features/canvas/CanvasBuildStages.tsx:98–104`
Completed stages use `line-through` decoration. Unconventional for a progress tracker — reads as deleted/error.
→ Use opacity/colour dimming instead.

**F-L8** `frontend/src/pages/muse/Overview.tsx:32–33`
`isBuilding` compound expression is hard to read and has implicit precedence. Extract into named booleans or add grouping parentheses.

**F-L9** `frontend/src/pages/muse/Research.tsx:162–163`
`processing` is counted only within already-filtered `user || approved` resources. Unapproved agent resources processing are not counted. Intentional but undocumented.

## Notes

- **B-H4 (chroma import):** `builder.py` imports `get_vector_store` from `vector_store/chroma.py` directly. This is the same pattern as `rag.py` (pre-v0.3). The factory function lives only in `chroma.py` and is not re-exported from `base.py`. The portability issue is pre-existing and not introduced by v0.3. Tracked separately.
- **B-M (muse creation KL build):** Triggering KL on Muse creation with no resources is intentional — the KL builds from description alone (the "thin Canvas" flow). This is correct per PRD §5.4.
